1. 控制器

  1. pod类型的资源，删除pod后，不会重建

  2. 替用户监视并保证相应的节点上始终有用户所期望的副本数量的pod在运行

  3. 如果所运行的pod副本数超过了用户期望的，那么控制器就会删掉，直到和用户期望的一致

  4. 如果所运行的pod副本数低于用户期望的，那么控制器就会创建，直到和用户期望的一致

4. StatefulSet控制器

有状态应用管理 StatefulSet
Stateflset(有状态集，缩写为sts)常用于部署有状态的且需要有序启动的应用程序，比如在进行 SpringCloud 项目容器化时，Eureka的部署是比较适合用 StatefulSet 部署方式的，可以给每个 Eureka 实例创建一个唯一且固定的标识符，并且每个Eureka 实例无需配置多余的 Service,其余 Spring Boot 应用可以直接通过 Eureka的 Headless Service 即可进行注册。

StatefulSet 的基本概念
Statefulset 主用于管理有状态应用程序的工作负载 API对象。比如在生产环境中，可以部署 ElasticSearch 集群、MongoDB 集群或者需要持久化的RabbitMQ集群、Redis 集群、Kafka集群和 ZooKeepgr集群等。和 Deployment 类似，一个 StatefulSet 也同样管理着基于相同容器规范的 Pod。不同的是,StatefulSet 为每个 Pod 维护了一个粘性标识。这些 Pod 是根据相同的规范创建的，但是不可互换，每个 Pod 都有一个持久的标识符，在重新调度时也会保留，一般格式为statefulseuName-Number。比如定义一个名字是 Redis-Sentinel的 StatefulSet，指定创建三个Pod，那么创建出来的Pod 名字就为 Redis-Sentinel-0、Redis-Sentinel-1、Redis-Sentinel-2。而StatefulSet 创建的 Pod般使用 Headless Service(无头服务)进行通信，和普通的 Service 的区别在于 Headless Service 没有ClusterIP，它使用的是 Endpoint 进行互相通信，Headless 一般的格式为:
statefulSetName-{0…N-1}serviceName,namespace.sve.clusterlocal.

  • serviceName为Headless Service 的名字，创建StatefulSet时，必须指定 Headless Service名称;

  • 0…N-1为Pod所在的序号，从0开始到N-1;

  • statefulSetName为StatefulSet的名字:

  • namespace 为服务所在的命名空间:

  • cluster.local为CusterDomain(集群域)

    ​ 假如公司某个项目需要在 Kubermetes 中部署一个主从模式的 Redis，此时使用 Statefulset 部署就极为合适,因为 Statefulset启动时,只有当前一个容器完全启动时,后一个容器才会被调度，并且每个容器的标识符是固定的，那么就可以通过标识符来断定当前Pod的角色。StatefulSet 的有序启动、稳定标识符、持久化存储和更好的有状态管理能力，使它成为部署 Redis 主从模式的理想选择。特别是在分布式系统中，确保主节点和从节点按照正确的顺序启动，并且每个节点的角色可以通过标识符来明确，这对 Redis 的高可用性和稳定性至关重要。
    ​ 比如用一个名为 照redis-ms的 Statefulset 部署主从架构的 Redis，第一个容器启动时，它的标识符为 redis-ms-0，并且Pod 内主机名也为redis-ms-0,此时就可以根据主机名来判断，当主机名为redis-ms-0的容器作为Redis的主节点，其余从节点，那么 Slave连接Master 主机配置就可以使用不会更改的 Master 的Headless Service，此时 Redis 从节点(Slave)配置文件如下:

为什么 StatefulSet 启动比较好?

  1. 有序启动和终止： StatefulSet 会保证 Pods 按照顺序启动和终止。当你创建 StatefulSet 时，它会按照编号顺序启动 Pod（如 redis-0, redis-1, redis-2 等），并且只有前一个 Pod 启动成功后，下一个 Pod 才会被调度。这对于 Redis 主从模式尤为重要，因为我们通常希望在从节点启动之前，确保主节点已经完全启动并可用。
  2. 稳定的网络身份： StatefulSet 中的每个 Pod 都有一个稳定的 DNS 名称。比如，Pod redis-0 可以通过 redis-0.redis.default.svc.cluster.local 来访问，这样即使 Pod 重启，它的 DNS 名称和网络身份都不会改变。这对于 Redis 集群中的主从模式非常重要，因为从节点需要知道主节点的固定地址以进行连接。
  3. 持久化存储： StatefulSet 会为每个 Pod 提供独立的持久化存储（通过 PersistentVolume）。对于 Redis 来说，这意味着每个实例（主节点或从节点）都会有独立的存储卷，即使 Pod 被重新调度或重启，数据也能保留下来，不会丢失。这对 Redis 的高可用性和数据持久化非常关键。
  4. 管理有状态服务： StatefulSet 还提供了对 Pod 状态的更好管理，确保有状态服务（如 Redis 主从集群）能够正确地进行负载均衡、扩展和缩减。通过 StatefulSet 部署 Redis，可以确保从节点和主节点之间的关系更加明确，并且在扩容或缩容时不会影响到服务的稳定性。
  5. 角色确定： 如你所说，StatefulSet 的 Pods 有稳定的标识符（例如 redis-0, redis-1 等）。通过这些标识符，可以直接识别每个 Pod 的角色，比如 redis-0 可以作为主节点，redis-1 和 redis-2 可以作为从节点。结合应用层的逻辑，可以方便地在应用启动时通过标识符确定每个节点的角色。

k8s里的有状态应用和无状态应用的区别

在 Kubernetes 中，有状态应用和无状态应用是两类应用的不同类型，它们在架构设计、部署方式、生命周期管理等方面有所不同。理解它们的区别有助于在 Kubernetes 上更好地设计和管理应用。

1. 有状态应用（Stateful Application）

有状态应用是指那些需要持久化存储、能保存状态信息的应用。它们通常依赖于某种形式的持久化存储，数据在多个生命周期中得以保存。

特征：

  • 持久化数据：有状态应用会生成需要在多个实例之间共享或跨重启持续存在的数据。这些数据通常存储在外部系统中（如数据库、文件系统等），也可以是内部的持久化存储（如 Kubernetes 中的 PersistentVolume）。
  • 稳定的网络标识符：每个实例通常会有一个固定的网络标识符（如 DNS 名称或 IP 地址），这使得其他应用可以依赖于这些固定标识符进行通信。Kubernetes 中，StatefulSet 可以确保每个 Pod 都有唯一且稳定的 DNS 名称。
  • Pod 状态管理：有状态应用的实例通常需要特定的顺序启动和停止。StatefulSet 通过确保 Pod 按照顺序启动（并且在有序停止时保持一致性）来管理这些应用。
  • 扩展性和故障恢复：有状态应用通常需要处理数据复制、备份和恢复等需求，尤其是在分布式数据库、缓存系统（如 Redis）等场景中。

示例：

  • 数据库：如 MySQL、PostgreSQL、MongoDB。
  • 缓存系统：如 Redis、Cassandra。
  • 消息队列：如 Kafka、RabbitMQ。

Kubernetes 资源：

  • StatefulSet：Kubernetes 中的 StatefulSet 是专门用来管理有状态应用的控制器。它提供有序的部署、稳定的网络标识符和持久存储支持。

2. 无状态应用（Stateless Application）

无状态应用是指那些不需要持久化存储或保持状态的应用。每个实例都是独立的，处理请求时不依赖于之前的任何状态信息。

特征：

  • 无持久化存储：无状态应用通常不会保存任何持久化的数据，或者保存的数据是可以丢失的。例如，Web 服务器、API 服务等。
  • 弹性伸缩：无状态应用可以非常灵活地扩展和缩减，因为每个实例都是独立的，不依赖于其他实例的状态。Kubernetes 可以根据负载自动增减副本数，且不需要担心数据一致性或状态问题。
  • 可以随意重启和替换：无状态应用的实例可以在不影响业务逻辑的情况下被随时替换或重启。由于没有依赖持久数据，重启或替换不会导致数据丢失或服务中断。
  • 负载均衡：无状态应用通常依赖负载均衡来分配请求，因为每个实例的状态是相同的，可以平等处理请求。

示例：

  • Web 服务：如 Nginx、Apache HTTP Server。
  • API 服务：如 RESTful API 服务、GraphQL 服务。
  • 微服务：许多微服务架构中的服务通常是无状态的。

Kubernetes 资源：

  • Deployment：Kubernetes 中的 Deployment 是专门用来管理无状态应用的控制器。它支持自动化部署、扩展、回滚等操作，非常适合无状态应用的生命周期管理。

3. 有状态应用 vs 无状态应用

特性 有状态应用 无状态应用
持久化存储 需要持久化存储（如数据库、日志） 无需持久化存储，数据是可丢失的
实例依赖性 每个实例有唯一的标识符，依赖其他实例的状态 实例独立，可以随时添加或删除
网络标识符 固定的网络标识符（DNS 名称、IP 地址） 不需要固定标识符，负载均衡即可
扩展与缩减 扩展和缩减时需要考虑状态的一致性 扩展和缩减非常简单，实例是无状态的
部署控制器 使用 StatefulSet 来管理实例顺序和状态 使用 Deployment 来管理实例
实例恢复与故障恢复 需要考虑数据一致性、备份和恢复 可以随时重启，实例之间不依赖状态

总结：

  • 有状态应用需要持久化存储、稳定标识符和顺序启动等机制，适用于数据库、缓存、消息队列等需要维护状态的服务。
  • 无状态应用不依赖持久化存储，实例可以随时重启和扩展，适用于 Web 服务、API 服务等无需保存状态的场景。

Kubernetes 通过不同的控制器（StatefulSet 和 Deployment）来支持这两类应用，帮助你管理和扩展这些应用。

depolyment也可以数据持久化啊，为什么不适合部署redis和es集群等

你说得对，Deployment 也可以配置持久化存储，并且可以用来部署一些需要持久化数据的应用，但它并不适合用来部署像 Redis、Elasticsearch 等需要高可用、分布式、状态管理和有序部署的集群应用。下面是为什么 Deployment 不适合用于这类应用的几个原因：

1. 无稳定的网络标识符

  • 在 Kubernetes 中，Deployment 管理的 Pod 是无状态的，Pod 的名称是动态分配的，并且可能会在每次重启时发生变化。

  • Redis 或 Elasticsearch 集群通常需要固定的网络标识符来确定节点的角色（比如主节点、从节点，或在 Elasticsearch 中的数据节点、协调节点等）。没有稳定的标识符就无法确保节点间的连接和同步。

    例如，Redis 主从节点的角色通常是根据 Pod 的稳定标识符（如 redis-0, redis-1）来确定的。Deployment 中的 Pod 名称是动态生成的，不保证顺序或稳定性，因此无法满足这种需求。

2. 无序的 Pod 启动和终止

  • Deployment 通过 ReplicaSet 管理 Pod，Pod 的启动和终止是无序的。Kubernetes 会同时启动所有副本，并没有严格的顺序，这对于有状态的分布式应用（如 Redis 和 Elasticsearch）来说是一个问题。
    • Redis 主从模式：Redis 要求先启动主节点，然后才能启动从节点，这样从节点才能连接到主节点进行复制。如果在 Pod 启动时没有有序性，可能会导致从节点无法及时连接到主节点，从而导致初始化失败。
    • Elasticsearch 集群：Elasticsearch 集群需要特定的节点类型（主节点、数据节点、协调节点等），这些节点类型和启动顺序可能会影响集群的健康状态。如果没有严格的启动顺序和稳定的节点角色，集群很可能无法正确初始化。

3. Pod 终止时的状态管理

  • StatefulSet 在 Pod 终止时会保证 Pod 的有序性，避免在删除 Pod 时出现不一致的状态。
  • Deployment 在 Pod 被终止或重启时不会保证有序，这对需要数据一致性和故障恢复的有状态应用来说非常重要。比如，在 Redis 集群中，若主节点被终止，必须确保新节点接管主节点角色后，其他节点才会跟随更新。

4. 持久化存储

  • Deployment 是可以配置持久化存储的，通常通过 PersistentVolumeClaim（PVC）来实现，但它本身并没有对 Pod 的持久化存储做出任何额外的管理，它只是简单地绑定存储卷。

  • StatefulSet 不仅提供了持久化存储支持，还确保了每个 Pod 都有独立的存储卷，并且在 Pod 重启或重新调度时，存储卷与 Pod 保持绑定。

    对于 Redis 或 Elasticsearch 这样的应用，Pod 的存储需要是独立且持久的。StatefulSet 可以确保每个 Redis 实例（如主节点和从节点）都绑定到各自的持久化存储卷。而 Deployment 中的 PVC 共享存储卷，无法做到这种级别的持久性和隔离性。

5. 故障恢复和数据一致性

  • StatefulSet 支持 Pod 的顺序部署、扩展、缩减、恢复等，确保有状态服务能够保持一致性。它会根据 Pod 的稳定标识符进行管理，并且在故障恢复时，StatefulSet 会优先恢复最关键的 Pod。
  • 对于 Redis 和 Elasticsearch 等有状态服务，如果使用 Deployment，Kubernetes 会随机选择一个 Pod 来重新调度，这可能导致数据丢失或集群状态不一致，特别是当集群内的数据需要分布式一致性时（例如在 Redis 的主从复制或 Elasticsearch 的分片分布上）。

6. 缩放问题

  • StatefulSet 支持有序扩展和缩减，这对某些应用来说至关重要。比如，当 Redis 集群的副本数增加时，新节点需要按照顺序加入集群，确保数据的正确复制。
  • Deployment 不保证 Pod 启动顺序，因此扩容时可能会导致 Redis 主从节点不按顺序启动，影响集群的初始化和数据同步。

总结

虽然 Deployment 可以通过 PersistentVolume 实现数据持久化，但它的设计理念和功能并不适合有状态的分布式应用，如 Redis 和 Elasticsearch。这些应用通常需要：

  • 稳定的网络标识符 来确定节点角色。
  • 有序的 Pod 启动与终止，确保服务的正确性。
  • 独立持久存储，确保每个节点的数据隔离。
  • 自动化的故障恢复和数据一致性。

因此，StatefulSet 是 Kubernetes 中专为有状态服务设计的控制器，能够更好地满足这些需求，确保 Redis 和 Elasticsearch 等集群的高可用性、数据一致性和故障恢复。
